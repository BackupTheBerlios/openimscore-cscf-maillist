Hi,

You are right about AVP vendor-id problem, but now I took a look at the 3gpp
specification 29.229 release 6 and 7 to see if the message s were in
conformance with the specifications and noticed that they are not in
conformance, for example the message format for RTR should be the following
:

 <Registration-Termination-Request> ::=                    < Diameter
Header: 304, REQ, PXY, 16777216 >

< Session-Id >

{ Vendor-Specific-Application-Id }

{ Auth-Session-State }

{ Origin-Host }

{ Origin-Realm }

{ Destination-Host }

{ Destination-Realm }

{ User-Name }

*[ Associated-Identities ]*

**[ Supported-Features ]*

**[ Public-Identity ]*

*{ Deregistration-Reason }*

*[ AVP ]

*[ Proxy-Info ]

*[ Route-Record ]




**

> I also made some tests with r76 on forced de-registration via HSS. These
> are
> my findings:
>
> If one sets the status of an IMPU to UNREGISTERED then the HSS sends a
> diameter PPR-packet to S-CSCF, which is not answered by S-CSCF. The PPR
> packet sent by the HSS is eventually corrupt (bad vendor ID length).
>
> If one sets the status of an IMPU to NOT REGISTERED then the HSS sends a
> diameter RTR-packet to S-CSCF, which is not answered by S-CSCF. The RTR
> packet sent by the HSS is eventually corrupt (bad vendor ID length).
>
> Besides above mentioned problems with eventually corrupt packets I have
> the
> question: What is the difference between setting the user_state to
> "UNREGISTERED" and "NOT REGISTERED"?
>
> Looking at the reaction of the HSS I think, that only setting to "NOT
> REGISTERED" makes sense.
