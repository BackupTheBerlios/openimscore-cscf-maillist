Yes, there are 2 service routes as we want to force any initial request
to go through the THIG node so that it can Record-route, etc and follow
the dialog in order to encrypt all needed headers.

Double encryption could be avoided by specifying a regular expression to
match on the old header in order to encrypt it. I think that I did
implement that :) some time ago, but not sure. Anyway, THIG, as specd,
should only encrypt headers from ones own network and this is why this
regexp checking is possible.

-Dragos
