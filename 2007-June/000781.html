Hi,

I have some questions regarding S-CSCF behavior.

[My recognition]
1.TS.24.229 specifies S-CSCF behavior to avoid
SIP loop routing.

2.OpenIMS realize the behavior by implementing
pushing identifier in route header when ifc routing is detected by S-CSCF.

3.OpenIMS is able to invoke multiple SIP-ASs sequentially.
Identifier that is inserted by S-CSCF indicate the progress of SIP AS
invocation.

[Question1]
Are there any 3GPP specifications that specifies multiple SIP-ASs
invocation?
(In another word, OpenIMS comply with some kind of specification regarding
this behavior?
Or are those things left 'implementation specific?')

[Question2]
As far as I could test in my environment, multiple SIP-ASs invocation can be
achieved by
OpenIMS though it seems it can not handle BYE signals properly.

For example in case Alice calls Bob.
Let's assume that IFC for Alice is set to route AS1,IFC for Bob is set to
route AS2.

SIP INVITE request is routed like below.
(180Ringing,200OK,ACK request is routed properly)
Alice ->OpenIMS->AS1->OpenIMS->AS2->OpenIMS->Bob.

However BYE request from Bob is not routed because S-CSCF returns '403
Forbidden'
when BYE request is transfered from AS2 to AS1 via OpenIMS(S-CSCF).

Is it a know issue?
