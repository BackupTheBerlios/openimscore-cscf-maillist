Hi,

 

We detected an issue with the ISC module triggering, when we have a user configured with two triggers (the triggers indicate that the INVITE request passes trough AS1 and AS2 and the CANCEL request passes only trough AS2) to two different Application Servers. With this type of trigger the behaviour is not quite what we expected, because the Cancel request will go to AS1 when it should only go to AS2. Apparently this happens because of the way the TM module controls the transactions. Even if the ISC module considers that the request should not go to AS1 and should go to AS2, this behaviour will not be applied. The Cancel transaction will be matched with the Invite transaction and then change the destination URI of the Cancel message previously filled in by isc_forward function. We currently don't know how to solve this issue, or if this is a real issue. To us it makes sense that this type of scenario could happen because we can define these types of triggers. Do you agree with this? What could be a solution for this issue?
