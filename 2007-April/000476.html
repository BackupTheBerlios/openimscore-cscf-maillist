Dragos,

I understand what you are saying, but this is not the case for me, in my case I have a Public User Identity registered with a implicitly registered Public User Identity set on the S-CSCF, all with the same contact. So what I'm trying to say is that I consider that the S-CSCF has an incorrect behaviour. When the expected behaviour would be to proceed with the de-registration of the corresponding implicitly registered Public User Identity set and send a SAR message after receiving a de-registration of a Public Identity,  this is not the behaviour the S-CSCF follows. What happens is that only the Public Identity of the de-registration is deregistered and not the implicitly registered Public User Identity set. 
Do you agree?

True, but I was saying that it is often the case, observed by us, that
the client disappears (loss of IP connectivity for example) and then
comes back again and registers with another contact on the same
identity. Then, when it turns off this new contact, a SAR is expected,
but this won't come until the first contact will expire. This is because
there was no information that the first identity actually is no longer
there. If you want to make sure that you deregister all your contacts
then send a de-register with a "Contact: *" header - but be aware that
if you have multiple phones on the same public identity, then the first
one doing a deregister with a Contact: * will practically de-register
all other phones!

-Dragos
