Hi John,

well, according to RFC3261, it does what it should do:

The branch parameter in the topmost Via header field of the request
   is examined.  If it is present and begins with the magic cookie
   "z9hG4bK", the request was generated by a client transaction
   compliant to this specification.  Therefore, the branch parameter
   will be unique across all transactions sent by that client.

Your server does put the cookie, but then it does not change the rest of
the id when it changes the transaction. We rely on that as it is unique
and if you support it, it's way easier to identify the transaction
quickly. If you say that it is the same transaction, then it is a
retransmission and it right to ignore it.

Here are all the differences between the messages.
1c1
< INVITE sip:+31356872696 at ims.xcsl.nl SIP/2.0
---
> INVITE sip:+31356872695 at ims.xcsl.nl SIP/2.0


But what happened there was a "fork". And in a fork a new branch is to
be created, which is practically a new transaction. You can't resend the
same transaction, because that was already answered and you'll just get
the answer retransmitted to you.

Hope this clears it...

I don't know about the old SIP RFC, but we're not interested in that...
