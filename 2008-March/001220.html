I think it is important. The trace only contains the response and not the
request which cause the response.

I guess the problem is probably a bad SIP URI of the request.
At least this is what the scscf.cfg says in the #Terminating branch. Look
into scscf.cfg.

Another hint to avoid the missed packets in the trace.
If you use a positive filter on ports you cannot catch fragmented packets,
because they do not contain port information. In this case use the inverse
(negative filter) and eliminate packets you do not need.

I use in case of packet-fragmentation the following command to trace all
packets (including fragments) without SSH and MySQL:
tcpdump -i any -s 3000 -w /var/tmp/tracefile.pcap not port 22 and not port
3306
